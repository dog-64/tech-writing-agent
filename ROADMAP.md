# Дорожная карта развития Tech Writing Agent

## Введение

Документ содержит предложения по расширению и улучшению функциональности агента для проверки технических текстов по методологии «Пиши, сокращай» Ильяхова.

Каждое предложение имеет:
- **Название** — понятное описание функции
- **Описание** — что это даст пользователю
- **Реализация** — как это можно сделать
- **Приоритет** — важность (🔴 высокий, 🟡 средний, 🟢 низкий)
- **Сложность** — затраты на реализацию (⭐ легко, ⭐⭐ среднее, ⭐⭐⭐ сложное)

## 0. ПРИОРИТЕТ: Интеграция книги "Ясно, понятно"

**Описание:**
Обработать и интегрировать материалы из книги Максима Ильяхова "Ясно, понятно. Как доносить мысли и убеждать людей с помощью слов" (FB2 формат) как дополнительный источник правил для анализа текста.

**Почему это первый приоритет:**
- Это продолжение методологии «Пиши, сокращай» от того же автора
- Расширяет возможности агента с новыми правилами убедительности и влияния
- Позволяет проверять не только стиль, но и эффективность аргументации
- Дополняет текущий агент новыми аспектами анализа

**Текущее состояние:**
Агент использует правила только из методологии «Пиши, сокращай».

**Реализация:**

### Этап 1: Парсинг книги (FB2)
- Скачать/получить книгу в формате FB2
- Парсить FB2 структуру (главы, параграфы, примеры)
- Извлечь все правила, советы, примеры
- Организовать в структурированный формат (JSON или YAML)

### Этап 2: Категоризация правил
Выделить из книги основные темы:
- **Убедительность** — как структурировать аргументы
- **Честность** — как не манипулировать читателем
- **Эмоции** — как вызвать нужные эмоции
- **Прозрачность** — как быть честным с аудиторией
- **Примеры** — когда и как использовать примеры
- **Рассказы** — как использовать истории для влияния
- **Доказательства** — какие доказательства убедительны

### Этап 3: Интеграция в агента
- Добавить новый раздел анализа: "Убедительность"
- Проверять каждый текст на соответствие правилам книги
- Давать рекомендации как сделать текст убедительнее
- Показывать примеры из книги когда применимо

### Этап 4: Документация
- Создать справку с правилами из книги
- Добавить примеры до/после для каждого правила
- Создать профиль "persuasive-writing" для убедительных текстов
- Добавить в README ссылку на книгу

**Примеры вывода после интеграции:**

```
📝 Анализ файла README.md

✅ МЕТОДОЛОГИЯ "ПИШИ, СОКРАЩАЙ":
  Текст в порядке, нет стоп-слов

🎯 УБЕДИТЕЛЬНОСТЬ (из книги "Ясно, понятно"):
  ⚠️  Мало примеров в разделе "Установка"
      Рекомендация: добавьте реальный пример использования

  ⚠️  Отсутствует история успеха
      Рекомендация: покажите кейс когда это помогло

  ✓ Структура честная и прозрачная
  ✓ Аргументы хорошо обоснованы

📊 ИТОГ:
  Стиль: 9/10
  Убедительность: 6/10
```

**Приоритет:** 🔴 🔴 КРИТИЧНЫЙ (выше всех остальных)
**Сложность:** ⭐⭐⭐ сложное (требует анализа книги и переработки агента)

**Зависимые задачи:**
- [ ] Получить книгу в FB2 формате
- [ ] Парсер для FB2 структуры
- [ ] Сортировка правил по категориям
- [ ] Расширение `tech-writing-agent.md` с новыми правилами
- [ ] Обновление `analyze-markdown.sh` для новых проверок
- [ ] Тесты на примерах из книги

**Результат:**
Агент будет проверять не только грамматику и стиль, но и **убедительность и эффективность аргументации** текста. Это даст пользователям возможность писать тексты которые не только красивы, но и по-настоящему воздействуют на аудиторию.

**Реализация:**

**Сложность:** ⭐⭐ среднее

```bash

```

### 1.3 Анализ читаемости

**Описание:**
Вычислять метрики читаемости текста: индекс Флеша, средняя длина предложения, сложность слов.

**Реализация:**
- Добавить функции в `analyze-markdown.sh` для подсчёта
- Вычислять индекс Флеша (Flesch Reading Ease)
- Подсчитывать среднюю длину предложения и слова
- Показывать предложения которые слишком длинные

**Приоритет:** 🟡 средний
**Сложность:** ⭐⭐ среднее

**Примеры вывода:**
```
📊 ЧИТАЕМОСТЬ:
  • Индекс Флеша: 72/100 (хорошо для технической документации)
  • Средняя длина предложения: 18 слов (норма: 15-20)
  • Самое длинное предложение: 48 слов (⚠️ слишком длинное)

💡 Рекомендация: Разбейте предложения на строках 15, 23, 31
```

## 2. Персонализация и конфигурация

### 2.1 Профили для разных типов текстов

**Описание:**
Создать предустановленные профили анализа для разных типов текстов: техническая документация, блог, письма, отчёты.

**Реализация:**
- Создать файл `~/.claude/writing-profiles.json` с предустановками
- Каждый профиль содержит: список стоп-слов, уровень строгости, метрики
- Пользователь может выбрать профиль при проверке

**Приоритет:** 🟡 средний
**Сложность:** ⭐ легко

**Примеры профилей:**
```json
{
  "profiles": {
    "documentation": {
      "stopwords": ["базисно", "по сути", "собственно"],
      "strictness": "high",
      "check_structure": true,
      "min_flesch_score": 60
    },
    "blog": {
      "stopwords": ["кстати", "в общем-то"],
      "strictness": "medium",
      "allow_colloquialisms": true,
      "min_flesch_score": 70
    },
    "report": {
      "stopwords": ["как правило", "безусловно"],
      "strictness": "high",
      "formal": true
    }
  }
}
```

### 2.3 Уровни строгости анализа

**Описание:**
Три режима анализа: мягкий (только критичные ошибки), средний (все проблемы), строгий (включая стилистические).

**Реализация:**
- Добавить переменную `STRICTNESS_LEVEL` в конфиг
- Мягкий: только стоп-слова + опечатки
- Средний: всё + многословие + неясность
- Строгий: всё + рекомендации по стилю + метрики

**Приоритет:** 🟡 средний
**Сложность:** ⭐ легко

**Использование:**
```bash
# В .claude/settings.json добавить:
"strictness": "medium"  # или "soft", "strict"
```

## 4. Интеграция с экосистемой

### 4.1 CI/CD интеграция

**Описание:**
Добавить поддержку автоматической проверки текстов в GitHub Actions, GitLab CI и других CI/CD системах.

**Реализация:**
- Создать GitHub Action для проверки PR
- Комментировать PR с результатами проверки
- Блокировать merge если найдены критичные ошибки
- То же для GitLab CI

**Приоритет:** 🔴 высокий
**Сложность:** ⭐⭐ среднее

**Пример GitHub Action (`.github/workflows/check-docs.yml`):**
```yaml
name: Tech Writing Check
on: [pull_request]

jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Run tech-writing-agent
        run: |
          for file in $(git diff --name-only origin/main | grep '\.md$'); do
            claude --include ~/.claude/agents/tech-writing-agent/tech-writing-agent.md "Проверь $file"
          done
```

- Проверить наличие ключевого слова в заголовке
- Подсчитать плотность ключевых слов
- Проверить длину описания (150-160 символов)
- Предложить улучшения для заголовков

**Приоритет:** 🟢 низкий
**Сложность:** ⭐⭐ среднее

**Пример:**
```
🔍 SEO АНАЛИЗ (ключевое слово: "Python")
  • Ключевое слово в заголовке: ✓
  • Плотность ключевого слова: 1.2% (норма: 0.5-2.5%)
  • Meta description: отсутствует (рекомендуется 150-160 символов)
  • Длина заголовка H1: 45 символов (хорошо)
```

## 7. Производительность и масштабируемость

### 7.1 Кэширование результатов

**Описание:**
Не проверять файлы которые не изменились, чтобы сэкономить API запросы.

**Реализация:**
- Сохранять хеш файла и результаты проверки
- При следующей проверке сравнивать хеш
- Если файл не изменился, использовать кэшированный результат
- Кэш хранить в `.claude/.writing-cache`

**Приоритет:** 🟡 средний
**Сложность:** ⭐⭐ среднее

---

### 7.2 Batch processing

**Описание:**
Возможность проверить все файлы проекта сразу, не по одному.

**Реализация:**
- Добавить команду `tech-writer --batch <dir>`
- Обойти все `.md` файлы в директории
- Собрать результаты в один отчёт
- Показать прогресс (файл 1 из 10...)

**Приоритет:** 🟡 средний
**Сложность:** ⭐⭐ среднее

**Использование:**
```bash
tech-writer --batch docs/
# Checking: docs/README.md (1/5)
# Checking: docs/INSTALL.md (2/5)
# ...
# Total: 5 files, 12 issues found
```

---

### 7.3 Локальный анализ (без API)

**Описание:**
Добавить простые проверки которые работают без обращения к Claude API.

**Реализация:**
- Regex проверка стоп-слов (встроенные)
- Встроенная проверка орфографии (hunspell)
- Подсчёт читаемости (алгоритм Флеша)
- API используется только для сложных проверок

**Приоритет:** 🔴 высокий
**Сложность:** ⭐⭐ среднее

**Преимущества:**
- Быстрее (нет задержки API)
- Дешевле (меньше API запросов)
- Работает оффлайн для базовых проверок
